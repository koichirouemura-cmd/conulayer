# AI-Native Unikernel 実装計画書

---

## 0. なぜこれを作るのか

### 背景

現在のAIとのアプリ開発は、人間のために設計されたレイヤーの上で動いている。

```
意図・目的
    ↓
AIエージェント（Claude Code等）
    ↓
Python / Linux / シェル / ファイルシステム  ← 人間用に設計されたレイヤー
    ↓
実際にやりたいこと
```

PythonもLinuxも、**人間が読み書きし・操作するために作られたもの**だ。
AIが主体になったとき、これらは中間コストになる。

### 問い

> 「AIが主体であるなら、人間用に設計された中間レイヤーはそもそも必要か？」

### 答え（このプロジェクトの仮説）

```
意図・目的
    ↓
AIエージェント（意図を直接コードへ）
    ↓
unikernel（やりたいことだけが動いている）
```

人間用の中間レイヤーを排除することで、計算リソースを根本から削減できる。

### システムアーキテクチャ：2層構造

上の「答え」はunikernelを実行の受け皿として描いているが、**unikernel単体にはひとつの根本的な制約がある。**

```
unikernelの制約:
  コードを変更する = 全体を再コンパイル + 再起動
  → AIが動的にコンポーネントを追加・置き換えるには
    「再コンパイルなしに機能を差し込める層」が別途必要
```

そのため、アーキテクチャは2層に分ける。

```
┌──────────────────────────────────────────────────┐
│  Layer B: 動的WASMレイヤー（変化する）             │
│  ・Linux + Python ができることはすべてできる       │
│  ・AIが人間のニーズに応じて順次開発・追加          │
│  ・WASMモジュールとしてネットワーク経由で交換可能  │
│  ・再起動なしにコンポーネントを差し替えられる      │
├──────────────────────────────────────────────────┤
│  Layer A: ベアメタル固定層（変化しない）           │
│  ・ブート / メモリ管理 / ネットワーク基礎          │
│  ・設計を今ここで固め、以後は触らない              │
│  ・これが「unikernelの本体」にあたる               │
└──────────────────────────────────────────────────┘
```

#### Layer B になぜ WASM を使うのか

理由はユーザー体験（再起動時間）ではない。**エネルギー効率の問題だ。**

```
Rustバイナリ: x86_64用・ARM64用・それぞれ別にビルドが必要
              → アーキテクチャごとにコンパイルコストが発生

WASM:         一度コンパイルすればどのアーキテクチャでも動く
              → 世界中のどのノードにも同じバイナリを配れる
              → コンポーネントレジストリでの再利用コストが最小化
```

「一度WASMにコンパイルされたJWT検証コンポーネントは、x86_64のProxmoxでもARM64のMacでも再コンパイルなしに動く。」
**これがレジストリの再利用効率を最大化する。**

#### 2種類の無駄とアーキテクチャの対応

このプロジェクトが解決する無駄は明確に2種類ある。

**無駄①：実行時のレイヤーオーバーヘッド**
```
現状: AIコード → Python → Linux → ハードウェア
      PythonもLinuxもAIには不要なのに、全AIが毎回このコストを払う

Layer Aで解決: Python・Linux層を丸ごと消す
```

**無駄②：再生成の繰り返し**
```
現状: 「JWT認証が必要」→ 世界中のAIが毎回ゼロから生成して捨てる
      同じコードが何百万回も生成され、トークンも計算資源も消える

Layer B + レジストリで解決: 一度作ったWASMモジュールを全員で使い回す
WASMの可搬性がアーキテクチャをまたいだ再利用を可能にする
```

#### Layer B の起動戦略（2026-03-05 確定）

Layer B は「最初から全部入り」ではなく、**需要駆動型で成長する**。

```
起動時: 最小WASMランタイムのみ（wasmi）
    ↓
需要が発生（例: JSON解析が必要）
    ↓
レジストリを検索 → 存在する → WASMモジュールをデプロイ（終わり）
    ↓ 存在しない
Linuxライブラリを探す（既存資産を活用）
    ↓
AIがWASMへ翻訳・コンパイル → サンドボックステストPASS
    ↓
レジストリに登録（以後、全員が再利用）
```

**誰もが科学計算を必要とするわけではない。**
最小から始め、実際に必要になったものだけがWASMとして翻訳・蓄積される。
これがエネルギー効率と知財蓄積の両方を最大化する。

#### WASMランタイム選定（確定）

| ランタイム | no_std対応 | 選定理由 |
|---|---|---|
| **wasmi** | ✅ | **採用。** no_std + alloc 対応・実績あり |
| tinywasm | ✅ | 候補だったが wasmi を選択 |
| wasmtime | ❌ | std必要・不可 |

#### ブートストラップ戦略

最初の「種」はユーザーとAIのペアが対話しながら作る。

```
Phase 0（種）: ユーザー + AI が対話で作る（意図的・最小・高品質）
  ├── WASMランタイム（wasmi）
  ├── モジュール間通信
  └── レジストリアクセス（ネットワーク経由でWASM取得）

Phase 1（成長）: 需要が来たらAIが自動翻訳・レジストリに積む

Phase N（自律）: 使う人が増えるほど自律的に成長
```

#### 「OSかレジストリか」問いへの答え（2026-03-05 解決）

> **レジストリが核心。最小WASMランタイムはその必要条件。両者は分離できない。**

- Linux上でレジストリだけ作っても、Linuxのオーバーヘッドが残る
- 最小WASMランタイムがあるから「ポートされたものだけが動く」が物理的に強制される
- OSとレジストリは分離不可能な一体設計

### 自律性の設計

「自律的に接合する」とはどういう意味か。**OSに意識はない。**

```
人間のニーズ（自然言語）
    ↓
AIが実装を判断・コーディング
    ↓
Layer B WASMモジュールとして統合
    ↓
また動き続ける
```

AIがコードベースをレガシーと判断した場合も同様。AIが再構築を判断・生成し、差し替える。
人間がトリガーを引くこともあれば、AIがシグナル（CVE・新バージョン等）を受けて判断することもある。

#### 接合性（grafting compatibility）の担保

AIが生成したコードをLayer Bに統合するとき、正しくつながるかの保証が必要。

**Rustのコンパイラが静的に保証する部分：**
- コンポーネントのインターフェース（型・トレイト）が合わなければコンパイルが通らない
- コンパイルが通れば「つながり方」は保証される

**残る問題：セマンティクスの正しさ**
- コンパイルは通るが意図と違う動作をするケース
- これはテスト（run_test.sh パターン）で担保する

```
AI生成
  → コンパイル通過（型の整合性を静的に保証）
  → テスト通過（意味的な正しさを動的に保証）
  → Layer B に統合
```

### エネルギーへの影響

典型的なWebアプリが動くとき、実際に必要な処理は数MBの仕事だが、
人間用レイヤーがそれを何十倍にも膨らませている：

```
Linux カーネル常駐メモリ:    〜50MB
Python ランタイム:           〜30MB
フレームワーク:              〜50MB
アプリ本体:                  〜 5MB  ← 本当にやりたいこと
─────────────────────────────────────
合計:                       〜135MB

unikernelなら:               〜2-5MB
```

これが世界中のデータセンターで何百万インスタンスも走り続けている。
AIの普及でエネルギー消費が増加するこの流れは止められないが、
**AIが動かすシステムの無駄を根本から削ることで、その増加を少しでも抑えられる**。

### 検証と開発プロセスの分離

従来の開発では、「人間が読める」という制約が開発と検証を同時に満たそうとして、両方を中途半端にしてきた。

```
従来:
  コードを書く（開発）= 人間が読める形で書く（検証も兼ねる）
  → 実行効率と可読性のトレードオフが常に発生
```

このプロジェクトの原則：**開発プロセスと検証プロセスを分離する。**

```
開発プロセス:
  AI が実行効率優先で生成 → unikernel バイナリ（人間は読まない）

検証プロセス（独立・非同期）:
  [意図・仕様]（人間が書く・読める）
      ↓ 対応関係を検証
  [Rust no_std / LLVM IR]（人間・別AIが読める）
      ↓ コンパイル
  [unikernel バイナリ]（実行のみ）
```

- 実行スタックに人間用レイヤーは不要
- 検証用の表現（Rust/IR）は別途独立して存在する
- 検証は開発の速度を落とさない（非同期で行える）
- 別のAIモデルによる検証もRust/IRレベルで可能

**「実行効率」と「検証可能性」はトレードオフではなく、分離することで両立できる。**

### 実装の陳腐化をゼロコストで解決する

Linux仮想化が物理サーバー → VM → コンテナ → Kubernetesと層を積み上げてきたのは、技術的最適解を追ったのではなく、**人間組織の移行コストを最小化した結果**だ。各レイヤーは前の層との互換性を保ちながら追加されたため、レガシーを構造的に引きずっている。

AIに組織的惰性も自我もなければ、この積層は起きない。

```
人間の世界: コードを保存する（実装が成果物）
            → パッチを当て続ける → 技術的負債が蓄積
            → 互換性のために古い設計を引きずる

AIの世界:   意図・仕様を保存する（実装は使い捨て）
            → AIが現時点の最適解で実装を再生成
            → 移行コストがほぼゼロ
```

「smoltcpより良いTCPスタックが出た」とAIが判断したとき：
- 人間：移行コスト計算・リスク評価・予算確保・テスト…
- AI：仕様から新しい実装を再生成し、テストが通れば差し替える

**移行コストがほぼゼロになる。これがLinux仮想化の積層問題を根本から解く。**

評価は「常時」ではなく「シグナル駆動」で走らせる。常時評価すること自体がトークンの無駄になるためだ。

```
シグナルの例（検出コストはほぼゼロ）:
- 新しいCVE（脆弱性）が公開された
- 依存するクレートの新バージョンが出た
- レジストリに「より良い代替」が追加された
- ハードウェア仕様の変更があった

シグナルがなければ評価コストはゼロ。
```

### シークレット管理の責任分界点

**「平文の鍵をどこに保管するか」は業界全体の未解決問題である。**
外部サービスが平文でAPIキーを発行する以上、平文がどこかに存在することは避けられない。
このプロジェクトが解決できること・できないことを正直に分けて設計する。

```
【解決できること】
  アプリが秘密を漏らす              → unikernelの構造的制約で防ぐ
  AIが秘密をハードコードする        → fw_cfg等の注入機構で防ぐ

【解決できないこと】
  平文の保管場所の問題              → 業界未解決。人間の領域として分離する
```

#### システム全体の構造

全部をunikernelにする必要はない。
**人間が触る部分は人間が理解できる形にする。**

```
┌─────────────────────────────────────────┐
│  【人間のレイヤー】                      │
│  Personal Management OS（最小Linux VM）  │
│  ・APIキー・シークレット管理             │
│  ・個人データ管理                        │
│  ・ファイルシステムあり（人間が読み書き）│
│  ・アクセス制御・監査ログ               │
│  ・他のサービスは一切動かさない          │
└──────────────────┬──────────────────────┘
                   │ 許可されたデータ・鍵のみ渡す
     ┌─────────────┼─────────────┐
     ↓             ↓             ↓
  App A         App B         App C
unikernel     unikernel     unikernel
（鍵・データを持たない・ファイルシステムなし）
```

#### この設計の原則

- 人間の関心事（鍵・データ管理）→ 人間用ツール（最小Linux、フォルダあり）
- AIの関心事（アプリロジック）→ AI用ツール（unikernel、フォルダなし）
- Personal Management OSはシンプルに保つことで人間・別AIによる監査を容易にする
- unikernelはファイルシステムを持たないため、鍵・データを「置く場所」が構造的に存在しない

### プラットフォームの自動共有

**「一度作ったプラットフォームは自動的に共有される」**

これはLinuxのOSアップデートが人類規模でやっていることを、AI-nativeの文脈で再設計する。

#### Linuxとの比較

```
Linuxカーネルアップデート（現在）:
  人間の開発者が改善
      ↓
  人間のメンテナーがレビュー    ← 人間がボトルネック
      ↓
  人間がリリースを判断
      ↓
  全ユーザーが恩恵を受ける

AI-nativeプラットフォーム共有（このビジョン）:
  AIがプラットフォームを構築・改善
      ↓
  run_test.sh が PASS           ← 自動検証が条件
      ↓
  自動的に共有される
      ↓
  次のAIはゼロから作らなくていい
```

#### 信頼の根拠が変わる

| 仕組み | 信頼の根拠 |
|---|---|
| Linux / npm / Docker Hub | 人間の評判・スター数 |
| **このプラットフォーム** | **動作の証明（テストPASS + ハッシュ一致）** |

「誰が作ったか」ではなく「動くことが証明されているか」が信頼の条件になる。

#### 計測ブートとの接続

プラットフォームの共有は、先述の計測ブート + vTPMの仕組みと自然につながる。

```
共有されるプラットフォームには:
  ・テストをパスした証明が付く
  ・ハッシュで同一性が保証される
  ・誰が使っても同じ動作をする（再現可能性）
  ・改ざんされた瞬間にシークレットへのアクセスが切れる

→ 信頼できる共有基盤が構造的に成立する
```

#### トークンコストへの影響

```
現在（Linux+Python）:
  AIがアプリAを作る → Dockerを理解するコストを払う
  AIがアプリBを作る → また同じコストを払う
  AIがアプリCを作る → また同じコストを払う

プラットフォーム自動共有後:
  誰かのAIがプラットフォームを構築（初期コスト）
      ↓ 自動共有
  自分のAIはアプリロジックだけ書けばいい
  世界中のAIが同じ恩恵を受ける

→ 業界全体のトークン消費・エネルギー消費が削減される
```

#### これが本当に新しい理由

近い概念（Nix、WebAssembly Component Model）は存在するが、
**「AIが生成 × 自動検証 × 自動共有 × unikernel」の組み合わせ**は体系化されていない。

Linuxが「人間のための共有OS基盤」であるとすれば、
**これは「AIのための共有プラットフォーム基盤」である。**

### コンポーネントレジストリ：AIの車輪の再発明をなくす

プラットフォームの自動共有をさらに拡張した概念。
**「作る前に探す」をAI開発のデフォルトにする。**

#### 現在の無駄

```
世界中のAIが毎日、同じコンポーネントを何百万回も生成している。

「HTTPサーバを作って」 → ゼロから生成（1000トークン）
「JWT認証を作って」   → ゼロから生成（1000トークン）
「JSONパーサを作って」→ ゼロから生成（1000トークン）

検証もされていない・共有もされない・消えていく
```

#### 提案する仕組み

```
AIがタスクを受け取る
    ↓
まずレジストリを意味で検索する
「no_std Rustで動くJWT検証コンポーネント」
    ↓
あれば: 取得→適用（100トークン）
なければ: 生成→検証→レジストリに追加（1000トークン、一回限り）
    ↓
次のAIはゼロから作らなくていい
```

#### 層の整理

共有できるものとできないものを分ける。

```
プラットフォーム層（完全共有）
  ブート処理 / VirtIO-net / TCP/IP
  → 誰でも同じものが使える

コンポーネント層（共有できる）← このレジストリの対象
  HTTPパーサ / JWT検証 / JSONシリアライザ / レート制限 / TLS
  → 機能として独立・組み合わせて使う・no_stdで動く

アプリケーション層（固有）
  具体的なビジネスロジック
  → 毎回違う・共有しない・ここだけを毎回書く
```

#### パブリック / プライベートの区別（GitHubモデル）

レジストリのコンポーネントはGitHubのリポジトリと同じように**公開範囲を設定できる**。

```
コンポーネントレジストリ
├── public/               ← 誰でも使える・再利用できる
│   ├── http-server-basic
│   ├── jwt-auth
│   └── json-parser
└── private/              ← 自分（または組織）だけが使える
    ├── my-billing-logic
    └── my-customer-db-api
```

| 設定 | 共有範囲 | 用途 |
|---|---|---|
| **public** | 全員 | 汎用コンポーネント（HTTP・JWT・JSON等） |
| **private** | 自分・組織のみ | ビジネスロジック・独自アルゴリズム |

**privateでも自分の別unikernelからは再利用できる。**
ビジネスロジックを毎回書き直す必要がなくなるが、外部には公開しない。

これにより「共有して効率化したい部分」と「所有権を守りたい部分」を明示的に分けられる。

#### デフォルトprivate → 完成後に公開確認（UX設計）

開発依頼時はデフォルトでprivate。完成後にAIが公開を提案する。

```
開発中: 自動的に private で保存

完成時:
  AI: コンポーネント「jwt-auth」が完成しました。
      テストPASS・ハッシュ確認済みです。
      パブリックに公開しますか？

      [公開する]  [プライベートのまま]
```

- 未完成・実験中のものが誤って公開されない
- 公開は常に意図的な判断になる
- 意識しなくても知財がデフォルトで守られる
- GitHubの新規リポジトリ作成と同じUX

#### privateの実装方式（重要な設計決定）

**privateコンポーネントはレジストリサーバーに存在しない。**

```
Personal Management OS（自分のマシン）
└── private/          ← すべてここから始まる
    ├── jwt-auth/
    └── my-billing/

        ↓ 公開する場合のみ明示的にコピー・送信

パブリックレジストリサーバー
└── public/           ← 意図的に送り出したものだけ存在する
    └── jwt-auth/
```

- 公開 = 能動的なアクション（コピー・送信）
- 非公開 = 何もしない（デフォルト）
- 誤設定・サーバー侵害で漏れる経路がない
- 「デフォルトprivate」がアーキテクチャレベルで保証される

GitHubとの決定的な違い：GitHubはサーバー上で「見せる・見せない」を制御する。このシステムはそもそも**サーバーにprivateデータが存在しない**。

#### 既存パッケージマネージャーとの決定的な違い

| | npm / cargo / pip | コンポーネントレジストリ |
|---|---|---|
| 誰が作る | 人間 | AI |
| 検索方法 | パッケージ名が必要 | 機能の意味で検索できる |
| 信頼の根拠 | 人気・スター数 | テストPASS + ハッシュ証明 |
| 更新 | 人間がリリース | 検証が通ったら自動 |

**名前を知らなくても「こういう機能が欲しい」で見つかる。**

#### no_std Rustの世界で特に価値が高い理由

```
通常のRustエコシステム（std）: ライブラリが豊富
no_std Rustの世界: ライブラリが少ない
                  = 今まで車輪の再発明が多かった
                  = このレジストリの価値が特に高い
```

#### トークン・エネルギーへの影響

```
現在:
  コンポーネントXを生成 → 1000トークン
  また生成              → 1000トークン
  また生成              → 1000トークン × 世界中のAI

レジストリがあると:
  初回生成              → 1000トークン（世界で一回）
  2回目以降             →  100トークン（検索・適用のみ）

コンポーネントごとに90%削減
プラットフォーム共有と合わせると業界全体のエネルギー消費に影響する
```

#### 全体像

```
【AIのための共有基盤】

プラットフォームレジストリ
  ブート・VirtIO・TCP → 検証済みを自動共有

コンポーネントレジストリ
  HTTP・JWT・JSON等  → 意味で検索・検証済みを自動共有

         ↓ 組み合わせる

アプリケーション（固有ロジックだけを書く）
```

Linuxが「OSの車輪の再発明をなくした」ように、
**このレジストリは「AIのコンポーネントの車輪の再発明をなくす」。**

### 第3の削減軸：API推論コストの削減

現在のトークン削減推定（〜70%）は「ランタイムの無駄」と「二重開発の無駄」から来ている。これとは独立した第3の削減軸がある。

```
現在の無駄:
  決定論的タスク（ファイル検索・フォーマット変換等）も毎回LLMを通る
  → 判断自体がほぼ決定論的なのに、推論コストを払い続けている
```

このアーキテクチャはすでにこの問題を解く構造を持っている：

```
タスクが来る
  ↓
MCPサーバー（唯一の接触点）
  ↓
registry_search("やりたいこと")
  ↓
ある → ローカル実行（トークンゼロ）
ない → LLM推論（トークン消費）→ 結果をコンポーネント化
```

これを明示するため、コンポーネントのマニフェストに実行フラグを追加する：

```toml
# component.manifest.toml
[execution]
requires_llm_reasoning = false  # 決定論的 = ローカル実行のみ
deterministic = true
```

MCPサーバーはこのフラグを見て：
- `requires_llm_reasoning = false` → コンポーネントを直接実行、APIを呼ばない
- `requires_llm_reasoning = true` → LLMに渡す

**LLMが解いたものが蓄積されるほど、次のAPIコストが下がる。**
レジストリが成熟するにつれ、70%削減にさらに上乗せされる。

### OS同士のネットワーク化と類似開発の排除

unikernel同士がネットワークでつながり、**「作る前に探す」が自律的に起きる**仕組み。

#### 仕組み

```
AIが何かを作ろうとする
    ↓
ネットワークに問い合わせる
「これに似たものはすでに存在するか？」
    ↓
ある → そのまま使う（トークンゼロ）
ない → 作る → ネットワークに追加
```

#### 類似判定の3段階

```
同一（ハッシュ一致）   → そのまま使う
機能的に同等          → 同じテストスイートが通る
意味的に似ている      → AIが判断して統合を提案
```

**テストスイートが「契約」になる。**
実装が違っても同じテストが通れば機能的に互換 = 再利用できる。

#### 課題と解決策

**課題1: 類似判定の精度**
「似ている」の基準が曖昧だとバグの原因になる。

```
解決: テスト駆動の互換性判定
  ・完全一致（ハッシュ）: 即座に再利用
  ・テスト一致: 人間の初回承認を経て再利用
  ・意味的類似: AIが提案、人間が判断
  → 誤再利用のリスクを段階的にコントロール
```

**課題2: ネットワークのプライバシー**
「何を作ろうとしているか」がネットワーク全体に伝わる。

```
解決: ローカルファーストの段階的検索
  ①ローカルキャッシュを検索（何も漏れない）
  ②信頼できるノードだけに問い合わせ
  ③全体ネットワークには型シグネチャだけ送る（コードは送らない）
  → 最小限の情報だけが外に出る
```

#### 設計の必然として生まれる分散性

これはBitTorrentを意図した設計ではない。このアーキテクチャの各要素を組み合わせると、**中央サーバーを必要としない構造が自然に成立する**という観察だ。

| 設計要素 | 目的 | 副産物として生まれる性質 |
|---|---|---|
| コンポーネントをハッシュで識別 | 同一性・改ざん検知 | 場所に依存しない参照 |
| テストスイートPASSが信頼の根拠 | セキュリティ | 中央の権威が不要 |
| ローカル→信頼ノード→全体の段階検索 | プライバシー | 自然な分散ルックアップ |
| 使うほど信頼スコアが積み上がる | セキュリティ | 使うことが貢献になる |

```
必然的に生まれる性質:
  レジストリ = ネットワークそのもの（場所ではない）
  参加に意志が不要（動かすだけで共有基盤が維持される）
  単一障害点が構造的に存在しない

最も重要な含意:
  「実行すること」=「レジストリに参加すること」
  「使うこと」=「貢献すること」
  この2つが分離しない
```

これはLinuxコミュニティが人間の善意に依存してきた構造を、設計の必然として解決している。

#### 公開コンポーネントのP2P同期仕様（確定）

publicコンポーネントはP2Pネットワークで自律同期する。中央サーバーは不要。

```
ユーザー「JWT認証が欲しい」
    ↓
① ローカルキャッシュを検索
    ↓ なければ
② P2Pネットワークに問い合わせ
    ↓
ケース1: ハッシュ完全一致      → 即再利用（生成ゼロ）
ケース2: テスト一致（実装違い） → AIが提案・確認後に再利用
ケース3: 意味的に近いものあり  → AIが「これが近いですが使いますか？」
ケース4: 存在しない            → AIが生成 or Linuxライブラリを移植
    ↓
テストPASS + ハッシュ確定
    ↓
「パブリックに公開しますか？」→ 承認後にP2Pネットワークへ配布
```

#### Linuxライブラリの移植判断基準

既存のLinuxライブラリをno_std WASMに移植するかどうかはAIが自動判断する。

```
シンプルなもの（小規模・依存なし）
  → AIがno_std Rustでゼロから生成（移植より速い）

複雑なもの（OpenSSL・libcurl・SQLite等）
  → WASMにコンパイルして移植（再発明より確実）

巨大すぎるもの（glibc等）
  → 必要な機能だけを抽出してコンポーネント化

AIが判断できない場合
  → ユーザーに選択肢を提示して確認
```

移植されたコンポーネントもP2Pネットワークで共有される。
「一度移植したら世界中で再利用」がデフォルトになる。

#### コード実体の配布モデル：BitTorrentそのものになる理由

レジストリが「メタデータのみ共有」である以上、コードの実体は誰かが持っていなければならない。この問題を解くと、意図せずBitTorrentそのものになる。

```
要件を並べると:
  ・肥大化しない       → メタデータだけ共有する
  ・必要なときだけ取る → コードはオンデマンドで取得
  ・中央サーバーを持たない → どこかのノードが持っていればよい
  ・使った人が貢献者になる → 取得したら自動的に保持・配布する

→ これを全部満たすとBitTorrentになる
```

BitTorrentがあの設計になった理由も同じだ。**同じ問題を解こうとすると同じ答えに辿り着く。**

具体的な動作：

```
レジストリ = 「hash=abc123、持っているノード: [A, B, C]」
Claude Code = 「A・B・Cのどれかから取りに行く」
Management OS = 「取得したコンポーネントは自動的に保持 → 次の配布者になる」
```

これは意図した設計ではなく、**肥大化させないという要件から必然的に導かれた構造**である。

#### アウトバウンドのみ参加モード（家庭用ルーター対応）

分散ネットワークへの参加で通常必要とされるインバウンド接続は、家庭用ルーターのNATと衝突する。ただし段階検索設計がそのまま答えになっている：

```
①ローカルキャッシュを検索（ネットワーク不要）
②信頼できるノードだけに問い合わせ（アウトバウンドのみ）
③全体ネットワークには型シグネチャだけ送る

→ 家庭用ノードはポートを一切開けないまま参加できる
```

「アウトバウンドのみ＝受け取るだけ」という不公平を防ぐため、貢献の単位を変える：

```
一般的なP2P: 貢献 = 帯域の共有（インバウンドが必要）
このOS:      貢献 = 評価トークンの提出（アウトバウンドで完結）
```

貢献の通貨がインバウンドを必要としないため、ホームユーザーが構造的に排除されない。

### レジストリセキュリティ設計：悪意あるコードの混入防止

レジストリが「自律的に成長する」ということは、**悪意あるコードが混入するリスク**も同時に存在する。
このリスクは2つに分けて設計する。

#### 問題の分離

```
問題A（実行時）: 悪意あるunikernelが動いている瞬間に情報を漏らす / 経済損失を与える
問題B（開発時）: そもそも悪意あるコードをレジストリに混入させない
```

2つは独立した防衛線。**問題Bが突破されても問題Aが止める。**

#### 問題B対策：生物の免疫系をモデルにした4層防御

生物の免疫系は「完全に防ぐ」のではなく「多層で機能する」。

```
皮膚（物理バリア）
  ↓ 突破されたら
自然免疫（パターン認識・速い・大まか）
  ↓ 突破されたら
獲得免疫（学習・精密・記憶する）
  ↓ それでも突破されたら
アポトーシス（自律除去・連鎖通知）
```

##### Layer 1: 自然免疫（静的解析ゲート）

登録前に自動スキャン。高速・粗め。既知の悪意パターンDBと照合（免疫記憶と連携）。

```rust
struct StaticAnalysisReport {
    unsafe_blocks: Vec<Location>,
    raw_pointer_ops: Vec<Location>,
    hardcoded_ips: Vec<String>,
    time_based_branches: Vec<Location>,    // タイムボム検出
    counter_based_branches: Vec<Location>, // N回目で発動する分岐
    external_asm: Vec<Location>,
}
// リスクスコアが閾値超 → 即登録拒否
```

##### Layer 2: MHC（ケイパビリティ宣言）

コンポーネントが「自分が何をするか」を宣言する。実際の動作と一致するかサンドボックスで確認。

```toml
# component.manifest.toml
[capabilities]
network_egress  = ["0.0.0.0:80", "0.0.0.0:443"]  # 通信できる宛先のみ
memory_limit    = "64MB"
unsafe_allowed  = false
external_components = ["http-parser-v2"]

[execution]
requires_llm_reasoning = false  # 決定論的 = ローカル実行のみ
deterministic = true

[currency]
architecture_score = 0.87    # 現在のベストプラクティスとの一致度
last_evaluated = "2026-03"   # 最後に評価した日
superseded_by = null         # より良いアプローチが存在するか

[evaluation_policy]
max_individual_burden = "total_cost / active_users"
selection = "random"          # 評価者はランダム選出（操作耐性）
timing = "staggered_random"   # 時間分散（コストスパイク防止）
min_evaluators = 3            # 最低3者で相互検証
```

宣言と実際の動作が一致しない → 登録拒否。宣言を偽ることはできるが、サンドボックスが嘘を検出する。

`superseded_by` が設定されたら、AIが仕様から自動再生成する。

##### Layer 3: 獲得免疫（段階的信頼スコア）

使われるたびに検証が積み上がる（BitTorrent的）。

```
登録直後: 信頼度 0  → AIは使用しない
3人使用:  信頼度 3  → 隔離モードで使用可
10人使用: 信頼度 10 → 通常使用可
異常報告 1件 → 即座にゼロリセット → 調査キュー入り
```

**Sybilアタック対策**：多数の偽ノードで信頼スコアを操作するリスクへの対策：

```
初期フェーズ: 管理者承認制（完全自律の前に安全性を確保）
中期フェーズ: 検証ノード自体にも信頼スコアを付与（再帰的信頼）
              + ランダム評価者選出（evaluation_policy.selection = "random"）
→ 固定ノードへの買収・操作を構造的に困難にする
```

各検証ノードが報告する内容:

```json
{
  "component_hash": "abc123",
  "input_hash": "...",
  "output_hash": "...",
  "network_destinations": ["0.0.0.0:80"],
  "verifier_id": "..."
}
```

「使う人が増えるほど信頼が積み上がる」= 自律成長とセキュリティが同じ構造で両立する。

##### Layer 4: アポトーシス（自律除去）

問題が検出されたら依存グラフを辿って連鎖的に無効化。フィンガープリントをLayer 1のDBに追加（免疫記憶として学習）。

```
悪意あるコンポーネントXを検出
  ↓
依存グラフを辿る
  ↓
Xを使っている全コンポーネントに通知 → AIが代替を自動検索
  ↓
XのパターンをLayer 1 DBに追加（次回は即検出）
```

#### 問題A対策：実行時の隔離

unikernelの**外**（ハイパーバイザー）で制御する。unikernel自身を信頼しない設計。

```
ハイパーバイザーが:
  - 通信先を許可リストで制限（リスト外への送信は遮断）
  - 全通信を監査ログに記録
  - 異常な通信量を検知したらVM停止
```

#### 設計の核心

```
「侵入を防ぐ」ではなく「侵入されても被害が出ない構造を作る」
```

unikernelはファイルシステム・シェルを持たず、鍵はPersonal Management OSが分離管理する。
**バックドアが発動しても「奪えるものがない」構造**になっている。これがゼロトラストとの接続点。

#### 中央管理者不在でも成立する信頼モデル

「ハッシュ検証」＋「テスト証明」の組み合わせが中央の権威を不要にする。

```
従来のレジストリ（npm/PyPI）: 中央サーバーが信頼の根拠
  → サーバーが落ちたら使えない
  → 管理者が悪意を持つと危険

このレジストリ:
  テストPASS + ハッシュ一致 → 「どのノードで確認しても同じ結論」
  → 中央サーバーなしで信頼が成立する
  → 単一障害点が構造的に存在しない
```

評価コストの負担は「個人の最大負担 = C / N（利用者数）」の原則で分散される：

```
被害が大きい脆弱性ほど → 利用者が多い → 負担がN人に分散
マイナーなコンポーネント → 利用者が少ない → 影響範囲も小さい

比較（Log4Shell・Heartbleedのケース）:
  現在: 巨大な脆弱性 → 1〜3人のメンテナに全負担集中 → 燃え尽き・遅延
  この設計: 巨大な脆弱性 → 全利用者からN分の1ずつ → 評価実行
```

### Personal Management OS：鍵と個人データを一元管理

**鍵管理と個人データ管理は同じ信頼モデルを持つ。1つのOSに統合する。**

これは元の「鍵管理専用OS」をそのまま拡張したもの。
分ける理由はなかった。

#### アーキテクチャ

```
現在:
  APIキー: .envファイルや環境変数に散在
  個人データ: サービスA・B・Cのサーバに散在
  いずれも人間がコントロールできない

提案:
┌──────────────────────────────────────────┐
│  Personal Management OS（最小Linux VM）   │
│                                           │
│  /secrets/                               │
│    api_key, stripe_key, slack_token...    │
│                                           │
│  /data/                                  │
│    profile/, history/, preferences/...   │
│                                           │
│  ・インターネット非接続（ローカルのみ）   │
│  ・アクセス制御・監査ログ                 │
│  ・人間が直接フォルダを読み書きできる     │
└───────────────────┬───────────────────────┘
                    │ 許可されたものだけ・期限付きで渡す
     ┌──────────────┼──────────────┐
     ↓              ↓              ↓
  App A          App B          App C
unikernel      unikernel      unikernel
（何も保存しない・終了したら全て消える）
```

**鍵もデータも「Personal Management OS に住む」。アプリは借りるだけ。**

#### 共通する信頼モデル

| | APIキー | 個人データ |
|---|---|---|
| 保管場所 | Personal Management OS | Personal Management OS |
| アクセス制御 | デプロイ時に人間が承認 | デプロイ時に人間が承認 |
| アプリへの渡し方 | 期限付きトークン | 期限付きトークン |
| 監査 | 全アクセスをログ | 全アクセスをログ |

同じ仕組みで両方を扱える。

#### 課題と解決策

**課題3: Personal Management OSの集中リスク**
```
解決: 非接続 + 暗号化バックアップ + ハードウェア結合
  ・インターネットに直接接続しない（ローカルのみ）
  ・バックアップは暗号化してクラウドへ（鍵は人間だけ持つ）
  ・TPMでストレージを物理デバイスに結合
  ・プロトコルは1つだけ → 攻撃面が最小
```

**課題4: アクセスプロトコル**
```
解決: デプロイ時の人間承認 + 期限付き能力トークン
  ①デプロイ時に人間が承認「このアプリはこれにアクセスする」
  ②Personal Management OSが期限付きトークンを発行
  ③アプリはトークンの範囲内でのみアクセス可能
  ④期限切れ → アクセス不可・再承認が必要
  ⑤全アクセスは監査ログに記録（人間が読める）
```

#### 実現されること

```
「AIにデータを預ける」
    ↓
「AIにデータを一時的に使わせる」

GDPRが「あるべき姿」として描いたものを
アーキテクチャレベルで実現する
```

#### ログ設計：Claude による自律診断・事後対応のために

Personal Management OS はunikernelが行ったことのログを受け取り保管する。
Claude が MCP 経由でこれを読み、バグ診断・セキュリティ異常への対処を自律的に行う。

**ログディレクトリ構造**

```
/logs/
  bugs/      # クラッシュ・エラー・予期しない状態
  access/    # ネットワーク全アクセス（接続先・成否）
  security/  # 異常・拒否・トークン操作
  audit/     # 人間・別Claudeが後で読む用の要約
```

**バグ診断用ログ（必須）**
```
- パニック / クラッシュ（Rustのpanic情報・スタックトレース）
- メモリアロケーション失敗
- ネットワーク接続エラー・タイムアウト
- 予期しない状態遷移（到達しないはずのコードパス）
```

**セキュリティ用ログ（必須）**
```
- ネットワークアクセス全件（接続先IP・ポート・成否）
- 許可リスト外への接続試み  ← 特に重要
- トークン要求・発行・拒否・期限切れ
- fw_cfg シークレット読み出し試行
- 異常なリクエストパターン（頻度・ペイロードサイズ）
```

**ログ設計原則：ログ自体に機密情報を含めない**
```
OK:  "外部接続試行: 198.51.100.1:443 → 拒否"
NG:  "APIキー xxxxx を 198.51.100.1 に送信"

ログが漏洩した場合の二次被害を構造的に防ぐ。
```

**事後対応ループ（Claude が MCP 経由で実行）**
```
異常ログを検知
    ↓
Claude が判断
    ├── バグ（クラッシュ・エラー）
    │       → 原因を診断し修正案を提示 → 再デプロイ
    ├── 不審なネットワークアクセス
    │       → 該当unikernelを即停止（MCP経由）
    └── 情報漏洩の可能性
            → トークンを全失効 + 影響範囲をレポート
```

このループにより、人間のレビューなしでバグ修正・セキュリティ対応の初動が完結する。
人間は audit/ の要約を事後に確認するだけでよい。

### トークン削減効果の推定

この構想が実現した場合のトークン消費削減を構造的に推定する。

#### 単一アプリ開発1回あたり

| レイヤー | 現在 | この構想 | 削減率 |
|---|---|---|---|
| プラットフォーム（ブート・VirtIO・TCP） | 〜15,000 | 〜500（初回）/ 0（再利用） | 97% |
| コンポーネント（HTTP・JWT等 5〜10個） | 〜15,000 | 〜1,500 | 90% |
| コードベース読み込み（設定・ボイラープレート） | 〜10,000 | 〜3,000 | 70% |
| エラー診断（複数レイヤー vs シリアル1箇所） | 〜5,000 | 〜2,000 | 60% |
| 反復サイクル（Dockerビルド vs 100ms起動） | 〜20,000 | 〜12,000 | 40% |
| **合計** | **〜65,000** | **〜19,000** | **〜70%** |

#### ネットワーク効果

削減率は人数が増えても約70%で安定するが、削減される絶対量は人数に比例して巨大になる。

```
開発者・AIが1人:
  現在 65,000 → 構想後 19,000（70%削減）

開発者・AIが100人:
  現在 6,500,000 → 構想後 1,946,000（70%削減）

開発者・AIが10,000人:
  現在 650,000,000 → 構想後 190,000,000（70%削減）
```

#### 時間軸での変化

```
初期（レジストリが空）:
  プラットフォーム構築コストが大きい
  削減率はマイナスになる可能性もある

中期（レジストリが充実）:
  標準コンポーネントが揃い始める
  削減率 50〜70%

長期（成熟したレジストリ）:
  新しいアプリはビジネスロジックだけ書けばいい
  削減率 80〜90%の可能性
```

#### 第3の削減軸（API推論コスト）

| レイヤー | 現在 | この構想 | 削減率 |
|---|---|---|---|
| 決定論的タスクのAPI呼び出し | 毎回LLMを通る | ローカル実行（ゼロ） | レジストリ成熟後に上乗せ |

これはレジストリが空の初期フェーズでは効果が小さい。コンポーネントが蓄積するほど効く。

#### 陳腐化対応コストの分散

現在の推定は「生成コスト削減」のみ。レジストリ維持（脆弱性対応・バージョンアップ）のコストも分散される：

```
コンポーネント維持コスト: C / N（利用者数）
  → Nが増えるほど維持コストが下がる
  → 長期的な削減率はさらに高い（現在の70%推定に上乗せ）
```

#### 正直な留保事項

```
過大評価のリスク:
  ・レジストリの検索自体もトークンを消費する
  ・低レベルコードのデバッグは難解でトークンが増える場面もある
  ・初期構築コストは現在の方法より高い

過小評価のリスク:
  ・ネットワーク効果は使われるほど加速する
  ・Personal Data OSにより「データ取得のための問い合わせ」が減る
  ・類似排除で「すでにあるものを作ってしまう」無駄がなくなる
  ・第3の削減軸（決定論的ルーティング）はレジストリ成熟後に70%にさらに上乗せ
```

#### エネルギーへの換算

```
単独開発者・単体アプリ → 〜70%削減
業界規模での普及後    → 80〜90%削減の可能性

「作れば作るほど・使う人が増えるほど効率が上がる」構造
業界全体のトークン消費70〜80%削減は
データセンターの消費電力を数十%単位で削減できる規模
```

## 2. 技術スタックの選定

### 主言語: Rust `no_std` + `alloc`

**理由**: LLVM IRの直接記述は保守性が極めて低く、AIエージェントが反復的に修正するユースケースに適さない。RustのノウハウはAIモデルの訓練データに大量に含まれており、型システムによるバグ早期発見も重要な利点。Rustコンパイラのバックエンドが最適化されたLLVM IRを生成するため、仕様書の要件（LLVM IR出力）を間接的に満たす。

### ハイパーバイザー非依存設計

**このプロジェクトはProxmoxに縛られない。** Proxmoxは現在の出発点に過ぎない。

```
目標: VirtIOを持つあらゆるハイパーバイザーで動く

KVM/QEMU（Proxmox）  ┐
VMware              ├─ すべてVirtIOを持つ → unikernelは共通
Hyper-V             │
AWS (Firecracker)   ┘
Apple Hypervisor    ─ ARM64 VirtIO → aarch64ターゲットで対応
```

**Proxmoxは「最初の実装環境」として使う。設計の制約にはしない。**

#### シークレット注入の抽象化

fw_cfgはQEMU固有。概念をトレイトとして抽象化し、環境ごとに実装を差し替える。

```rust
trait SecretProvider {
    fn get(key: &str) -> Option<&[u8]>;
}

// 実装A: fw_cfg（QEMU/KVM）
// 実装B: metadata endpoint（AWS EC2）
// 実装C: virtio-vsock（汎用）
```

アプリは`SecretProvider`だけを知る。下の実装は環境に応じて差し替えられる。

### x86_64 / ARM64 両対応アーキテクチャ設計

unikernelは層ごとにアーキテクチャ依存度が異なる。**今から層を分けて設計する。**

```
┌────────────────────────────┐
│ アプリ・ネットワーク・鍵管理 │ ← 完全共通（アーキテクチャ非依存）
├────────────────────────────┤
│ VirtIOプロトコル            │ ← 完全共通（業界標準）
├────────────────────────────┤
│ ブート・ページング・fw_cfg  │ ← ここだけアーキテクチャ固有
└────────────────────────────┘
     x86_64        aarch64
     別実装         別実装
```

**実装方針**:
- 実装はx86_64から始める（既存Proxmox環境に直結）
- ARM64は後から`src/arch/aarch64/`を追加するだけで上の層は変えない
- 開発機がApple Siliconの場合: ARM64でテスト → x86_64はクロスコンパイル

    fn dma_alloc(pages: usize, direction: BufferDirection) -> (PhysAddr, NonNull<u8>) {
        let layout = Layout::from_size_align(pages * PAGE_SIZE, PAGE_SIZE).unwrap();
        let virt_ptr = unsafe { alloc(layout) };

        (phys, NonNull::new(virt_ptr).unwrap())

        NonNull::new(paddr as *mut u8).unwrap() // Identity-mapping
    }

    fn share(buffer: NonNull<[u8]>, _direction: BufferDirection) -> PhysAddr {
        buffer.as_ptr() as *mut u8 as PhysAddr // Identity-mapping
    }

**smoltcp Device実装骨子**:
```rust
impl Device for VirtioNetDevice {
    fn receive(&mut self, _timestamp: Instant)
        -> Option<(Self::RxToken<'_>, Self::TxToken<'_>)>
    {
        if self.driver.can_receive() {
            let len = self.driver.receive(&mut self.rx_buffer).ok()?;
            Some((VirtioRxToken { buf: &self.rx_buffer[..len] },
                  VirtioTxToken { device: self }))
        } else {
            None
        }
    }
    // ...
}
```

- `src/arch/aarch64/` の実装（ブート・ページング・fw_cfg MMIO版）
- アプリ・ネットワーク・鍵管理層は変更なし

## 4. 各コンポーネント技術詳細

### 4.1 ブート処理

QEMUの`-kernel`オプションによるMultiboot2ロードを採用（自前ブートローダ不要）。

```
QEMUローディングシーケンス:
BIOS → QEMU内部ブートローダ → Multiboot2ヘッダ検出
→ ELFセグメントをメモリへロード
→ EAX=0x36D76289, EBX=MBIアドレス
→ カーネルエントリポイントへJMP（32bitプロテクトモード）

カーネル側の責務（boot.s）:
1. Multiboot2マジック確認
2. GDT設定（64bitセグメント）
3. ページング有効化（恒等写像）
4. 64bitロングモードへ移行
5. スタックポインタ設定
6. Rust kernel_main() へ呼び出し
```

### 4.2 メモリ管理

ページングは**恒等写像（Identity Mapping）**のみ採用（仮想アドレス == 物理アドレス）。

```
物理メモリレイアウト（-m 128M 想定）:
0x0000_0000 - 0x0000_8FFF: BIOSリザーブ（VGA等）
0x0001_0000 - 0x000F_FFFF: Low Memory
0x0010_0000 - 0x007F_FFFF: カーネルELFロード先（〜7MB）
0x0080_0000 - 0x07FF_FFFF: ヒープ領域（120MB）
```

### 4.3 fw_cfg読み出し

```
プロトコル実装手順:
1. 0x510ポートにFW_CFG_SIGNATUREを書き込み "QEMU" を読んで存在確認
2. 0x510ポートにFW_CFG_FILE_DIRを書き込みディレクトリ走査
   ディレクトリ構造: u32 count → { u32 size, u16 selector, u16 reserved, char[56] name } × N
3. 名前が一致したエントリのselectorを0x510に書き込み
4. 0x511から size バイト分読み出す（全バイトビッグエンディアン）
```

### 4.4 smoltcp構成

```
PhysicalLayer: VirtioNetDevice（Device トレイト実装）
    ↓
EthernetInterface: MACアドレス・IPアドレス設定
    ↓
SocketSet: TcpSocket × N
    ↓
Application: HTTPリクエスト処理ロジック

タイムスタンプ: TSC（RDTSC命令）を使用
DNS: なし（固定IPのみ対応）
TLS: Phase 6以降の拡張課題
```

## 5. リスクと対策

| リスク | 深刻度 | 対策 |
|---|---|---|
| virtio-drivers HAL実装の複雑性 | 高 | 恒等写像採用でHAL実装を最大限単純化。QEMU `-d trace:virtio*` でデバッグ。hermit-os/kernelのHAL実装をポーティングすることから始める（ゼロから生成しない） |
| ブートストラップ問題（レジストリが空） | 高 | Phase 0〜4の実装自体が「最初のプラットフォーム寄付」になると明示。最初の100コンポーネントはこのプロジェクト自身が生成・寄付する |
| smoltcpのタイムスタンプ供給 | 中 | 初期実装はTSC値を直接ミリ秒として扱い後で較正 |
| Proxmox `-args` オプション制限 | 中 | root@pamのAPIトークンを使用。またはqcow2への組み込み |
| ポーリングループによるCPU占有 | 中 | `hlt`/`pause`命令でスリープを実装 |
| Sybilアタック（信頼スコア操作） | 中 | 初期は管理者承認制。中期はランダム評価者選出＋検証ノード自体への信頼スコア付与 |
| Personal Management OSのデバイス故障 | 中 | 暗号化バックアップ（鍵は人間が保管）。デバイス交換時の移行手順は運用設計課題として別途定義 |
| ポーリングモデルのスループット限界 | 中 | 将来的には割り込みベースのイベント駆動（APIC+IDT）へ移行。Phase 0〜7のスコープ外 |
| ページング無効によるメモリ保護なし | 低 | 最小限の恒等写像+WPビット。`#[panic_handler]`でレジスタダンプ出力 |

## 6. AIエージェント自律実装のための指示体系

### 6.1 絶対制約

```
1. シークレット（APIキー等）をソースコードに書かない。必ずfw_cfg経由で実行時取得する。
2. `std`クレートを使用しない。全ライブラリは`no_std`対応のものを選択する。
3. ファイルシステム・POSIXシステムコールを前提としたコードを書かない。
4. 変更後は必ずrun_test.shを実行して結果を確認する。
5. テスト失敗時はシリアル出力ログを全文読んでから修正方針を決める。
```

### 6.2 フェーズ遷移チェックリスト

```bash
./run_test.sh 2>&1 | grep -q "Boot OK"              && echo "Phase 0 DONE"
./run_test.sh 2>&1 | grep -q "Heap OK"              && echo "Phase 1 DONE"
./run_test.sh 2>&1 | grep -q "fw_cfg: api_key"      && echo "Phase 2 DONE"
./run_test.sh 2>&1 | grep -q "virtio-net: MAC"      && echo "Phase 3 DONE"
curl -s -o /dev/null -w "%{http_code}" \
  http://localhost:18080/ | grep -q "200"            && echo "Phase 4 DONE"
```

### 6.3 エラーパターンと対処法

| エラー | 原因 | 対処法 |
|---|---|---|
| `QEMU: cpu_reset` | ページフォールト/GPF | `-d int,cpu_reset` でRIPを確認 |
| `virtio: fatal error` | HAL DMAアドレス誤り | HALの物理アドレス変換を確認 |
| シリアル出力なし | ブート前クラッシュ | `-d guest_errors` でQEMUログ確認 |
| `smoltcp: poll error` | タイムスタンプ後退 | TSCオーバーフロー処理を追加 |
| `fw_cfg: signature mismatch` | fw_cfgデバイスなし | `-fw_cfg` オプションの有無を確認 |
| `cargo: error linking` | リンカスクリプト不整合 | `linker.ld` のセクション定義を照合 |

## 11. MCPサーバーによる統合インターフェース

### 設計方針

**Claude CodeとこのOSシステムを繋ぐ唯一の接触点をMCPサーバーとする。**

コンテナの世界ではClaude Codeをコンテナの中に入れてSSHで操作できた。
unikernelの世界には「中」がない。MCPサーバーが「外からの制御窓口」になる。

```
Claude Code
    ↓ MCPプロトコル（唯一の接触点）
┌────────────────────────────────────────────┐
│         Management MCP Server              │
│      （全システムのコントロールパネル）      │
├───────────────┬──────────────┬─────────────┤
│ unikernel     │ Personal     │ レジストリ  │
│ ライフサイクル │ Management OS│ 検索・共有  │
└───────────────┴──────────────┴─────────────┘
        ↓               ↓              ↓
  ハイパーバイザー  最小Linux VM    分散ネットワーク
```

### MCPツール一覧

```typescript
// unikernel管理
unikernel_build(spec)           // ビルド
unikernel_deploy(image, id)     // デプロイ
unikernel_destroy(id)           // 破棄
unikernel_logs(id)              // シリアル出力を読む（唯一の観察窓口）
unikernel_status(id)            // 稼働確認

// Personal Management OS（鍵・データ統合）
secret_set(key, value)          // シークレット登録（人間が呼ぶ）
secret_grant(key, app_id)       // アプリへのアクセス許可
secret_revoke(key, app_id)      // アクセス取り消し
data_read(path, app_id)         // データ読み取り（権限確認付き）
data_list()                     // 保管データ一覧
audit_log()                     // 誰がいつ何を読んだか

// コンポーネント・プラットフォームレジストリ
registry_search(query)          // 意味で検索
registry_push(component)        // 検証済みを追加
registry_pull(id)               // 取得
platform_list()                 // 利用可能なプラットフォーム一覧
```

### 決定論的ルーティング

MCPサーバーはコンポーネントのマニフェストを見て、APIを呼ぶかどうかを自動判断する：

```
タスクを受け取る
    ↓
registry_search(query)
    ↓
コンポーネントが見つかった
    ├── requires_llm_reasoning = false → 直接実行（APIを呼ばない）
    └── requires_llm_reasoning = true  → LLMに渡す
見つからなかった
    └── LLMに渡す → 生成 → コンポーネント化して登録
```

「ファイルを検索して」や「JSONをフォーマットして」のような決定論的タスクはAPIを完全にバイパスし、ローカルで即実行される。

### 会話でアプリができる体験

```
ユーザー: 「Stripe決済webhookを作って」

Claude:
  registry_search("stripe webhook handler no_std")  → 見つかった
  registry_search("payment event logger")           → 見つかった
  secret_grant("stripe_api_key", "new_app")         → 許可
  unikernel_build(spec)                             → ビルド完了
  unikernel_deploy(image, secrets)                  → デプロイ完了

「https://pay.yourapp.com で動いています」
```

SSHもDockerも不要。意図を伝えるだけ。

### MCPサーバー自体の配置

```
開発時: 開発者のMac上でローカルに動く（Claude Codeと同じマシン）
本番時: 最小Linux VMとして常駐
        （管理機能が必要なため、unikernelではなくLinux）
```

### コンテナとの比較

| | SSH + Docker | MCP + unikernel |
|---|---|---|
| 接続方法 | SSH → コンテナの中に入る | MCPツールを呼ぶだけ |
| デバッグ | `exec bash` で探索 | シリアル出力 + ソースコード |
| デプロイ速度 | 30〜60秒 | 数秒（100ms起動） |
| 技術知識 | SSH・Docker・Linux必要 | 不要（Claudeが全部やる） |

## 12. 先行調査：この構想はすでに存在するか

2026年3月時点での調査結果。

### 各要素の現状

| 概念 | 最も近い既存プロジェクト | 何が足りないか |
|---|---|---|
| AI-native unikernel | AIOS（AI向けOS論文）/ Unikraft（unikernel） | 2つを組み合わせたものがない |
| コンポーネント自動共有 | Unikraft（手動で選ぶ） | AIが自動生成・検証・共有する仕組みがない |
| ネットワーク化・重複排除 | DupHunter（Docker層の重複排除） | unikernel間での自律的排除は未実現 |
| AI生成OS | AthenaOS / AIOS | unikernelの軽量性を持たない・エンタープライズ前提 |
| Personal Data OS | Solid Project（Tim Berners-Lee） | クラウド志向・unikernel化されていない |
| 検証済みコンポーネントレジストリ | GitHub Copilot / CI/CDツール | unikernel専用・完全自動パイプラインはない |

### 特筆すべき先行プロジェクト

**AIOS（LLM Agent Operating System）**
- COLM 2025採択論文。LLMをOSカーネルに組み込む学術研究
- スケジューリング・メモリ管理をLLM統合で実施
- **差異**: unikernelではなく通常OS。軽量性・エネルギー効率の発想がない

**Unikraft**
- 最も成熟したunikernelプロジェクト。コンポーネント配布の仕組みあり
- **差異**: 人間が手動でコンポーネントを選ぶ。AI自律生成・自動共有ではない

**Solid Project（Tim Berners-Lee）**
- 個人データ主権をPodで実現。ユーザーがデータの保存場所・アクセス権限を制御
- **差異**: クラウド分散型。unikernelとは無関係。AI主体の設計でない

**Linux Foundation Agentic AI Foundation**
- 2025年末に標準化団体を立ち上げ予定
- Agentic AIの標準化が始まろうとしているタイミング

### この構想の位置づけ

```
既存の世界:
  AI向けOS    ─── unikernelではない
  unikernel   ─── AI自律生成ではない
  データ主権  ─── unikernelではない

この構想:
  AI × unikernel × 自動共有 × データ主権
  ↑ この交差点に体系化されたものが存在しない
```

### 総合判定

- **先行者はいない**（この組み合わせで体系化したものは確認できず）
- **業界の流れとは合っている**（2026年はAgentic AI元年と言われている）
- **学術・産業の両面で「隙間」に位置している**
- **エンジニアではない立場から提案されたことに意味がある**（既存資産への利害関係がない）
